# Design — Per-System & Per-Policy Compliance Trends (0.7.2)

**Status:** approved, implementation pending
**Target release:** 0.7.2
**Author:** Yariv + Claude (design review)
**Last updated:** 2026-06-09

---

## 1. Goal

The dashboard's compliance trend is fleet-wide only. When a number moves, the
next question is always *"which host / which policy, and since when?"* — and
today there is no per-entity history to answer it.

Add hourly compliance snapshots **per system and per policy**, displayed as a
trend chart on the system live report (`/systems/report/{id}`) and the policy
live report (`/policies/report/{id}`), with a per-tenant retention setting.

Decisions locked with the user:
1. **Hourly granularity** — consistent with the existing fleet trend tick.
2. **Retention is a setting** — per-tenant "how long to keep", with the
   existing `*_retention_days` pattern (0 = keep forever).
3. **Scope: systems AND policies** — one table, one recorder, two charts.
4. **Display: live report pages** (not the systems-list, not saved snapshots —
   both can be added later without schema change).

---

## 2. Data model

### New table `entity_compliance_history` (schema v37)

```sql
CREATE TABLE entity_compliance_history (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    tenant_id   VARCHAR(191) NOT NULL DEFAULT 'default',
    entity_type TEXT NOT NULL,              -- 'system' | 'policy'
    entity_id   INTEGER NOT NULL,           -- systems.id / policies.id
    check_date  DATETIME DEFAULT CURRENT_TIMESTAMP,
    score_test   REAL,                      -- per-test axis
    score_strict REAL,                      -- all-or-nothing axis:
                                            --   system → score_policy
                                            --   policy → score_system
    tests_passed INTEGER,                   -- raw tallies for tooltips
    tests_failed INTEGER,
    FOREIGN KEY (tenant_id) REFERENCES tenants (id) ON DELETE CASCADE
);

CREATE INDEX idx_entity_history_lookup
    ON entity_compliance_history (tenant_id, entity_type, entity_id, check_date);
```

Design notes:

- **Both compliance axes are stored per snapshot** (`score_test` +
  `score_strict`). This is the 0.6.6 dual-mode lesson: when the tenant flips
  the Per-test / Per-system / Per-policy toggle, the chart re-renders the
  *entire* history in the chosen mode instead of showing an artificial step.
- **`score_strict` naming** — one column serves both entity types; the
  semantics follow `entity_type` (a system's strict axis is per-policy, a
  policy's is per-system). Documented here and at the recorder.
- **Not-scanned entities are skipped, not stored** — rows are only written for
  entities with `compliance_score >= 0`. No `-1` placeholder rows; a gap in
  the chart line is rendered as a gap (Chart.js `spanGaps: false`).
- **`entity_id` is not a foreign key** — deleting a system/policy must not be
  blocked by history, and history for deleted entities is reaped by retention
  plus an explicit delete in the entity-delete handlers (§5).
- The index covers the chart query exactly (`WHERE tenant_id=? AND
  entity_type=? AND entity_id=? ORDER BY check_date`) and makes the
  retention prune (`check_date < ?`) a range scan — same sargable pattern as
  the 0.5.4 prune fix.

### Storage math (hourly)

1,000 active systems + 100 policies × 24/day ≈ 26K rows/day:
- 90-day default retention ≈ **2.4M rows** — fine for SQLite with the index.
- Retention 0 (forever) is the operator's informed choice; the settings help
  text states the growth rate.

## 3. Recorder

Extend the existing **hourly tick** in the scheduler main loop (the same place
`record_compliance_history` runs). New helper, same file:

```
record_entity_history(pool)
```

Two **set-based** INSERTs (no per-row loop — important for big fleets):

```sql
INSERT INTO entity_compliance_history
    (tenant_id, entity_type, entity_id, score_test, score_strict,
     tests_passed, tests_failed)
SELECT tenant_id, 'system', id, score_test, score_policy,
       tests_passed, tests_failed
FROM systems
WHERE status = 'active' AND compliance_score >= 0;

INSERT INTO entity_compliance_history
    (tenant_id, entity_type, entity_id, score_test, score_strict,
     tests_passed, tests_failed)
SELECT tenant_id, 'policy', id, score_test, score_system,
       systems_passed, systems_failed
FROM policies
WHERE compliance_score >= 0;
```

The stored aggregates are already maintained by every recalc
(`update_system_stats` / `update_policy_stats`), so the recorder reads — it
never computes. Failure is logged and non-fatal to the tick (matches
`record_compliance_history`).

## 4. Retention

Two separate per-tenant settings (decided in review — item history is
high-volume and short-lived; the fleet total is cheap and audit-relevant
long-term), both following the existing retention conventions (`0` = keep
forever):

| Setting | Covers | Default |
|---|---|---|
| **`entity_trend_retention_days`** | `entity_compliance_history` (per-system / per-policy snapshots) | **90** |
| **`fleet_trend_retention_days`** | `compliance_history` (the dashboard's total trend — today it grows forever) | **365** |

- **Settings UI** — two fields next to the other retention inputs:
  "System/Policy Trend Retention (days)" and "Compliance Trend Retention
  (days)", each with the growth-rate note in the help text.
- **Prune** — both added to the existing daily maintenance tick alongside
  report/notification/container retention:

```sql
DELETE FROM entity_compliance_history
WHERE tenant_id = ? AND check_date < ?      -- Rust-computed cutoff (sargable)

DELETE FROM compliance_history
WHERE tenant_id = ? AND check_date < ?
```

- Each writes the same style of `retention.*` audit row with the removed
  count. Bringing `compliance_history` under retention is a deliberate
  behaviour change called out in the changelog (existing installs default to
  365 days; set `0` to restore keep-forever).

## 5. Lifecycle

- **System/policy deleted** → handler also deletes its history rows
  (`DELETE FROM entity_compliance_history WHERE tenant_id=? AND
  entity_type=? AND entity_id=?`). Cheap via the lookup index, keeps the
  table honest without FK coupling.
- **System re-enrolled with a new id** → new line; old rows age out via
  retention. (Same behaviour as containers.)
- **Tenant deleted** → `ON DELETE CASCADE` via the tenant FK.

## 6. UI

### System live report (`/systems/report/{id}`)

A "Compliance Trend" card above the policy cards:

- Handler fetches the last N points **mode-aware**, mirroring the dashboard:
  `system_compliance_mode == "policy"` → plot `score_strict`, else
  `score_test`. Flipping the toggle re-renders the whole line (no jump).
- Chart.js line chart (already loaded on report pages? — reuse the dashboard's
  setup; the canvas + dataset pattern is copied from
  `dashboard.html`'s `complianceTrendChart`).
- Tooltip shows score + `tests_passed`/`tests_failed` for that snapshot.
- Fewer than 2 points → card hidden (no sad single-dot chart).
- Time axis: render the raw hourly points; the chart's x-axis tick density
  handles long ranges (no server-side bucketing in v1).

### Policy live report (`/policies/report/{id}`)

Identical card; mode = `policy_compliance_mode` (`"system"` →
`score_strict`).

### Range selector (v1: fixed)

v1 ships with a single range — everything inside retention (the chart decimates
visually). A 7d/30d/90d selector is a v1.1 query-param addition; no schema
impact.

## 8. Non-goals (v1)

- Systems-list sparklines (render-cost question; needs its own pass).
- Freezing trends into saved/archived snapshots.
- Per-container trend history (containers churn; revisit with k8s work).
- Server-side downsampling / rollups — revisit only if retention-0 tenants
  report chart slowness.
- Alerting on trend movement (separate feature; this table is its natural
  data source later).

## 9. Resolved questions (review, 2026-06-09)

1. **List-page links to the chart?** No — the report pages are enough in v1.
2. **Retention defaults** — split into two settings: per-item history
   (`entity_trend_retention_days`) defaults to **90 days**; the fleet total
   (`fleet_trend_retention_days`) defaults to **365 days** (§4).
