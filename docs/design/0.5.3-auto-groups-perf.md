# Design — Auto-Group Reconciliation & Compliance Recalc Performance (0.5.3)

**Status:** implemented
**Target release:** 0.5.3
**Last updated:** 2026-06-02
**Predecessor:** `0.5.2-auto-groups.md`

---

## 1. Why

0.5.2 shipped automatic group assignment with a deliberately simple
reconciliation path: every heartbeat called `apply_auto_groups`
unconditionally, and any membership change triggered a **global**
`recalculate_current_compliance` across all tenants. Correct, but with three
avoidable inefficiencies on the hot path and one blast-radius problem on
multi-tenant deployments. This pass closes all four without changing
observable behaviour (same memberships, same compliance numbers).

---

## 2. The four changes

### P1 — Rules-first short-circuit (heartbeat hot path)

**Before:** `apply_auto_groups` loaded the full system snapshot — including a
scan of the `containers` table — *before* checking whether the tenant had any
rules. Installs that don't use auto-groups (the majority, especially early)
paid a container scan + membership query on every heartbeat for nothing.

**After:** the function takes the already-loaded rule slice. When there are
**zero** enabled rules, a single indexed `EXISTS` on `systems_in_groups`
(auto-managed only) decides whether there's any stale membership to clean up.
If not — the common case — it returns before loading the snapshot.

**Correctness:** we only skip when *both* the rule set and the system's auto
memberships are empty. If a rule was disabled out-of-band leaving stale
memberships, the EXISTS catches it and the empty target set removes them.

### P2 — Parse rules once per sweep (admin save path)

**Before:** `apply_auto_groups_for_tenant` looped over N systems, each
re-running `load_enabled_rules → parse_conditions → regex compile`. O(N×R) in
regex compilation on the synchronous path an admin waits on after saving a
rule.

**After:** the sweep parses the rule set **once** and reuses the parsed slice
across all N systems via `apply_auto_groups_with_rules`. Now O(R) compile +
O(N) eval.

### P3 — Fingerprint short-circuit (heartbeat hot path)

**Before:** even when a system's metadata was byte-identical to the previous
heartbeat, the full reconciliation ran (snapshot + rule eval + membership
query + diff).

**After:** `systems.auto_group_fp` stores a SHA-256 of the rule-relevant
fields at last evaluation (folded into the snapshot SELECT — no extra read).
On the heartbeat path, if the freshly-computed fingerprint equals the stored
one, neither the target set nor membership can have changed, so rule eval +
the membership query + writes are skipped. The fingerprint is re-stamped
whenever it drifts.

**Fields in the fingerprint:** hostname, ip, os, arch, ver, status,
mem_total_mb, disk_total_gb, containers_exists, runtimes, container_images.
**Deliberately excluded:** `uptime_secs` — it changes every heartbeat and
would defeat the short-circuit. A rule on `uptime_secs` reconciles on the
once-per-tick cadence rather than instantly; acceptable because uptime is
monotonic between reboots and crosses any threshold exactly once.

**The sweep bypasses P3.** When *rules* change (not fields), fingerprints are
unchanged but re-evaluation is required. The sweep passes
`check_fingerprint = false` to force it, and — critically — **nulls every
fingerprint in the tenant up front**. This restores the pre-P3 self-healing
property: if the inline sweep errors on a system (rare DB hiccup, that one
transaction rolls back), its fingerprint stays NULL and the next heartbeat
re-evaluates it. Successfully-swept systems get their fingerprint re-stamped
immediately, so steady-state skipping resumes.

### Scope #4 — Per-tenant compliance recalc

**Before:** `recalculate_current_compliance` recomputed **every** test, system,
and policy across **all** tenants. A single new system joining a group in
tenant A forced a full recompute of tenants B…Z. (Pre-existing — the manual
`sync_tx` edit path and startup share it — but auto-groups added a trigger
that fires far more often.)

**After:** the four aggregation passes take an optional `tenant` filter.
`recalculate_current_compliance(pool)` stays global (startup + sync_tx).
`recalculate_current_compliance_for_tenant(pool, id)` scopes to one tenant.
TASK F now collects the **DISTINCT dirty tenant_ids** and recalculates only
those.

**Correctness:** tenant isolation is a hard boundary in the schema — every
`results` / `systems` / `policies` / `tests` row carries `tenant_id` and all
aggregation JOINs are tenant-local. So per-tenant scoping yields identical
numbers to the global pass for that tenant; it just skips touching everyone
else.

**Why not scope to individual dirty *systems*?** The test/policy aggregations
are cross-system within a tenant (a policy's score depends on all its
systems). Scoping to individual systems would require rewriting the
correlated subqueries with per-system filters that don't break the
cross-system rollups — higher risk for a marginal gain over per-tenant. Tenant
granularity captures the multi-tenant blast-radius win, which is the actual
problem, at low risk.

---

## 3. Cost summary

| Path | Before | After |
|---|---|---|
| Heartbeat, no auto-rules | snapshot (incl. container scan) + membership query | 1 indexed EXISTS |
| Heartbeat, rules + unchanged fields | snapshot + rule eval + membership query + diff | snapshot + 1 hash compare |
| Heartbeat, rules + changed fields | full reconcile | full reconcile (+ fp stamp) |
| Rule save sweep | O(N×R) regex compile | O(R) compile + O(N) eval, + 1 bulk fp-null |
| Membership change → recalc | all tenants × (tests+systems+policies) | only dirty tenants |

---

## 4. Schema

`v28 → v29`: adds `systems.auto_group_fp TEXT` (NULL default → every system
treated as "changed" on its first post-upgrade heartbeat, reconciled once,
fingerprint populated thereafter). Idempotent `column_exists` guard; fresh
installs get the column in `initialize_database`.

---

## 5. Tests

`auto_groups::tests` grows to 26 cases. New for this pass:
- `fingerprint_is_stable_and_ignores_stored_fp` — same fields → same hash, and
  the stored `auto_group_fp` does not feed the hash.
- `fingerprint_changes_on_each_rule_relevant_field` — every fingerprinted
  field flips the hash (guards against forgetting to add a new field to the
  fingerprint when extending the `Field` enum).
- `fingerprint_ignores_uptime` — uptime_secs must NOT affect the hash.

The DB-backed reconciliation paths (P1 EXISTS bail, P3 skip, sweep null-first)
are exercised end-to-end by the existing integration flow; the pure-logic
fingerprint contract is unit-tested as above.

---

## 6. Not done (future)

- **System-level recalc scoping** — see §2 Scope #4 rationale. Would need the
  aggregation subqueries reworked to per-system filters preserving the
  cross-system policy rollup. Separate design pass if per-tenant proves
  insufficient at very large single-tenant scale.
- **Cached compiled rules across heartbeats** — P2 removed per-system
  recompilation within a sweep, but each heartbeat still parses+compiles the
  tenant's rules once. A process-level cache keyed on a rules-version would
  remove that, at the cost of cache-invalidation plumbing. Not worth it until
  profiling shows rule compilation is material.
