# Design — Automatic Group Assignment (0.5.2)

**Status:** approved, implementation pending
**Target release:** 0.5.2
**Last updated:** 2026-05-31

---

## 1. Goal

Eliminate the manual "drag this new system into the right group" click that admins do today. An admin defines a small set of **rules** ("hostname starts with `web-` and OS is Linux → put in `linux-web-prod` group"), and from then on every newly-enrolled or metadata-changed system lands in the right groups automatically — without further intervention.

---

## 2. Architectural decision — change-driven, deferred recalc

The naïve implementation ("evaluate all rules on every heartbeat, recompute compliance whenever group membership changes") would melt the server. Three observations defuse it:

1. **Rule evaluation is microseconds** — regex/CIDR/substring checks against fields already in the `systems` row. The expensive thing is *compliance recalc*, not rule eval.
2. **~99% of heartbeats don't change anything rule-relevant.** A system's hostname, OS, architecture, total memory etc. don't change between heartbeats. The 5-min `last_seen` bump alone shouldn't trigger group re-evaluation.
3. **Compliance recalc is already scheduled.** `recalculate_current_compliance` runs on its own tick (hourly today). Group churn can flow through the existing job rather than driving a per-heartbeat recalc.

### Resulting flow

```
agent ── heartbeat ──> /send handler
                          │
                          ├── UPDATE systems SET ...
                          │
                          ├── ingest containers ...
                          │
                          ├── ┌── compare old vs new {hostname, ip, os, arch,
                          │   │   ver, mem_total_mb, disk_total_gb, plus the
                          │   │   set of (runtime, image) pairs on this host}
                          │   │
                          │   │       no change → continue (no extra work)
                          │   │
                          │   │       changed → apply_auto_groups(system_id)
                          │   │                  ├── load tenant's rules
                          │   │                  ├── evaluate each
                          │   │                  ├── diff against current
                          │   │                  │   systems_in_groups rows
                          │   │                  ├── INSERT/DELETE as needed
                          │   │                  └── if anything changed:
                          │   │                       set systems.compliance_dirty=1
                          │   │
                          │   └── continue
                          │
                          └── sign + return response
                                          │
                                          ▼ (separately, on its existing tick)
                            existing scheduler
                            recalculate_current_compliance
                              picks up systems.compliance_dirty=1,
                              recomputes affected policies, clears bit.
```

**Key properties of this design:**

- **Zero extra heartbeat cost in the no-change case** (one struct comparison)
- **Bounded cost in the change case** — O(R) where R = rule count per tenant (typically <20)
- **No new scheduler task** — leverages the existing hourly compliance recalc
- **Instant grouping at enrollment** — a brand-new system trivially "changes from nothing" so its first heartbeat triggers evaluation
- **No data-driven coupling between heartbeat and compliance recalc** — they communicate through a dirty bit, not synchronously

---

## 3. Privacy stance

**No new data is captured.** Every rule parameter is already collected for an unrelated, legitimate operational reason — heartbeat metadata, the existing Systems list display, container inventory the admin already sees.

Explicit non-goals (consistent with the "privacy first" project moto):

- ❌ No process inventory shipped on heartbeat
- ❌ No installed-package inventory
- ❌ No environment-variable or runtime-config capture
- ❌ No compliance-status as a rule parameter (also circular)

## 4. Parameter catalog

Every parameter below is either already in `systems` / `containers` or trivially derivable from one of those:

| Parameter | Type | Source | Example |
|---|---|---|---|
| `hostname` | string | `systems.name` | `equals 'mail'`, `regex '^web-prod-\d+$'` |
| `ip` | string + CIDR | `systems.ip` | `in_cidr '192.168.10.0/24'`, `starts_with '10.'` |
| `os` | string | `systems.os` | `contains 'Ubuntu'`, `regex '(?i)debian'` |
| `os_family` | enum | derived from `os` | `equals 'linux'`, `in ['linux','freebsd']` |
| `arch` | enum | `systems.arch` | `equals 'x86_64'`, `in ['aarch64','arm64']` |
| `platform` | string | `systems.platform` (derived) | `equals 'x86_64-linux'` |
| `ver` | string | `systems.ver` | `equals '0.5.2'`, `>= '0.5.0'` (semver compare) |
| `status` | enum | `systems.status` | `equals 'active'` |
| `mem_total_mb` | number | telemetry | `>= 16000` |
| `disk_total_gb` | number | telemetry | `>= 500` |
| `uptime_secs` | number | telemetry | `< 86400` (recently rebooted) |
| `containers_exists` | boolean | derived from `containers` table | `true` |
| `has_runtime` | enum | derived | `equals 'docker'`, `in ['docker','podman']` |
| `any_container_image` | string | derived from `containers.image` | `contains 'nginx'` |

Operators per type:

| Type | Operators |
|---|---|
| string | `equals`, `not_equals`, `contains`, `not_contains`, `regex`, `starts_with`, `ends_with` |
| numeric | `eq`, `ne`, `lt`, `le`, `gt`, `ge` |
| IP | `in_cidr` (plus all string ops) |
| boolean | `equals true`, `equals false` |
| enum | `equals`, `not_equals`, `in`, `not_in` |
| semver-string (`ver`) | string ops + `>=`, `<=`, `>`, `<` (compared as semver) |

## 5. Data model

One new table:

```sql
CREATE TABLE auto_group_rules (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  tenant_id   VARCHAR(191) NOT NULL,
  group_id    INTEGER NOT NULL,                  -- target system_groups.id
  name        TEXT NOT NULL,                     -- "Production Web Servers"
  description TEXT,
  conditions  TEXT NOT NULL,                     -- JSON array, see below
  enabled     INTEGER NOT NULL DEFAULT 1,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (tenant_id) REFERENCES tenants(id)      ON DELETE CASCADE,
  FOREIGN KEY (group_id)  REFERENCES system_groups(id) ON DELETE CASCADE
);
CREATE INDEX idx_auto_group_rules_tenant ON auto_group_rules(tenant_id, enabled);
```

`conditions` is JSON: `[ { "field": "...", "operator": "...", "value": ... }, ... ]`. AND-only across conditions for v1.

One column added to `systems`:

```sql
ALTER TABLE systems ADD COLUMN compliance_dirty INTEGER NOT NULL DEFAULT 0;
```

Bit set to `1` whenever a group-membership change occurs that could affect this system's compliance scope; cleared by `recalculate_current_compliance` when it processes the system.

## 6. Rule schema (JSON in the `conditions` column)

```json
[
  { "field": "os_family",            "operator": "equals",      "value": "linux" },
  { "field": "hostname",             "operator": "regex",       "value": "^web-prod-\\d+$" },
  { "field": "ip",                   "operator": "in_cidr",     "value": "10.10.0.0/16" },
  { "field": "containers_exists",    "operator": "equals",      "value": true },
  { "field": "mem_total_mb",         "operator": "ge",          "value": 16000 }
]
```

Server-side validator on save:
- `field` must be in the catalog table
- `operator` must be valid for `field`'s type
- `value` parseable per type (e.g. CIDR for `in_cidr`, valid regex for `regex`, etc.)
- Regex is anchored as-given (caller writes `^...$` if they want full match)

## 7. Heartbeat integration

```rust
// In the /send handler, after the existing UPDATE systems ... SET ...:
let metadata_changed = match (&old_row, &new_row) {
    (Some(o), Some(n)) =>
        o.hostname != n.hostname ||
        o.ip       != n.ip       ||
        o.os       != n.os       ||
        o.arch     != n.arch     ||
        o.ver      != n.ver      ||
        o.mem_total_mb  != n.mem_total_mb ||
        o.disk_total_gb != n.disk_total_gb,
    _ => true,  // first heartbeat — always evaluate
};

// Containers can also affect rules (`containers_exists`, `has_runtime`,
// `any_container_image`). Set a flag if the ingest changed anything.
let containers_changed = ingest_containers(...)?;

if metadata_changed || containers_changed {
    apply_auto_groups(&mut tx, tenant_id, system_id).await?;
}
```

`apply_auto_groups`:

```rust
pub async fn apply_auto_groups(
    tx: &mut sqlx::Transaction<'_, sqlx::Sqlite>,
    tenant_id: &str,
    system_id: i32,
) -> Result<(), sqlx::Error> {
    let rules = load_enabled_rules(tx, tenant_id).await?;
    let snapshot = load_system_snapshot(tx, tenant_id, system_id).await?;

    let target_groups: HashSet<i32> = rules.iter()
        .filter(|r| r.matches(&snapshot))
        .map(|r| r.group_id)
        .collect();

    let current_auto_groups: HashSet<i32> = load_auto_group_membership(tx, system_id).await?;

    let to_add    = target_groups.difference(&current_auto_groups).copied().collect();
    let to_remove = current_auto_groups.difference(&target_groups).copied().collect();

    if to_add.is_empty() && to_remove.is_empty() { return Ok(()); }

    upsert_system_group_membership(tx, system_id, to_add, to_remove).await?;
    mark_dirty(tx, system_id).await?;
    Ok(())
}
```

`SystemSnapshot` holds the rule-input fields (hostname, ip, os, os_family derived, arch, platform, ver, status, telemetry, container summary). Built fresh from the DB on each `apply_auto_groups` call to avoid stale data races with concurrent heartbeats.

## 8. Distinguishing auto-managed memberships from manual ones

Critical invariant: when an admin manually drops a system into a group, an auto-rule that doesn't match shouldn't yank it out. Solution: track which memberships are auto-managed.

Option A — flag column on `systems_in_groups`:
```sql
ALTER TABLE systems_in_groups ADD COLUMN auto_managed INTEGER NOT NULL DEFAULT 0;
```

`apply_auto_groups` only INSERT/DELETEs rows where `auto_managed=1`. Manual rows (`auto_managed=0`) are immune. UI shows a small "auto" badge on auto-managed rows so admins know which can be edited manually.

Option B — separate `systems_in_groups_auto` table. Cleaner separation but duplicates the join logic everywhere. Rejected in favour of A.

---

## 9. Admin triggers

| Trigger | What runs |
|---|---|
| Heartbeat with rule-relevant change | `apply_auto_groups(system_id)` for that one system |
| Admin creates a new rule | Full sweep: `apply_auto_groups(system_id)` for every system in the tenant. Admin sees a brief progress indicator. |
| Admin edits an existing rule | Same — re-evaluate every system. |
| Admin disables a rule | Full sweep — same code path; the disabled rule no longer matches, so systems get removed from the group if they were auto-added by it. |
| Admin deletes a rule | Same as disable, then delete the rule row. |
| Admin deletes a group | FK CASCADE deletes its `auto_group_rules`; manual `systems_in_groups` rows go with the group (existing behaviour). |

The full-sweep cost (one admin action, run once) is O(N×R) where N=systems in tenant, R=rules. For a tenant with 1000 systems and 20 rules, that's 20,000 in-memory rule evaluations + a small number of INSERTs/DELETEs — sub-second.

## 11. Migration path

Schema migration v27 → v28:

```sql
CREATE TABLE auto_group_rules (...);
CREATE INDEX idx_auto_group_rules_tenant ON auto_group_rules(tenant_id, enabled);
ALTER TABLE systems ADD COLUMN compliance_dirty INTEGER NOT NULL DEFAULT 0;
ALTER TABLE systems_in_groups ADD COLUMN auto_managed INTEGER NOT NULL DEFAULT 0;
```

No data movement needed — every existing `systems_in_groups` row keeps `auto_managed=0` (manual), so the new feature doesn't disturb any existing membership.

## 12. Audit trail

New audit event types:

- `auto_group.rule_create` — admin defined a new rule
- `auto_group.rule_update` — admin edited a rule (incl. enable/disable)
- `auto_group.rule_delete` — admin removed a rule
- `auto_group.system_added` — a system joined a group via rule match (details: rule id, group id)
- `auto_group.system_removed` — a system left an auto-managed group (rule no longer matched)

The membership events fire only on actual change, not on every heartbeat. Detail field includes which rule triggered the change so an admin can answer "why did host X end up in group Y" without guessing.

## 13. MVP cut for 0.5.2

**In:**
1. `auto_group_rules` table + the `auto_managed` / `compliance_dirty` columns (schema v28)
2. `apply_auto_groups` function + change-detection hook in the heartbeat handler
3. Full-sweep on admin rule create/update/delete
4. Admin UI: list, create, edit, delete, enable/disable
5. "auto" badge on group chips in the systems list
6. Audit events for rule changes + membership changes
7. Compliance recalc integration via `compliance_dirty` (existing scheduler reads it)

**Deferred:**
- OR semantics (v1 is AND-only)
- Rule priority / order-sensitive matching
- Tags / labels as a rule parameter (would need a new `system_tags` table — out of scope until clearly needed)
- "Apply rule now" admin button — full sweep happens automatically on save; an explicit re-apply could come later if useful
- Rule preview ("show me which systems would match if I add this condition") — could be a future enhancement; for v1 the count badge updates after save

---

## 14. Open questions

1. **Rule conflict resolution.** Two rules targeting different groups can match the same system — that's fine, the system goes in both. Do we want anti-rules ("system in group X *cannot* be in group Y")? Probably not for v1 — admins can model the same intent with a single rule per group.

2. **Sub-second freshness.** The current design has up to one heartbeat interval of lag between a metadata change and the group update. Acceptable? Or do we want an admin-triggered "re-evaluate this system now" button on the system detail page?

3. **Performance ceiling.** Are there tenants with > 10,000 systems × > 50 rules where the full-sweep on rule edit becomes user-visible? If so, run it as a background job instead of inline. Probably fine to start inline; revisit if measurements show a problem.

---

## 15. Non-goals

- This is **not** a generic policy engine. The condition set is intentionally narrow — operationally relevant fields only.
- Rule evaluation is **not** transactional across systems. A rule applied to system A's heartbeat doesn't affect system B until B itself heartbeats (or an admin triggers a sweep). This matches operator intuition ("rules apply to each system independently") and keeps the model simple.
- We **do not** retroactively reassign manual memberships. If an admin manually put host X in group Y, no rule will ever remove it. Auto and manual coexist on the same `systems_in_groups` table distinguished by `auto_managed`.
