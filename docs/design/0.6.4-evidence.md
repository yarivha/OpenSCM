# Design — Test Result Evidence ("why did it fail?") (0.6.4)

**Status:** implemented
**Target release:** 0.6.4 (SaaS 0.4.4)
**Last updated:** 2026-06-07

---

## 1. Goal

Answer "**why did this client fail this test — and which part?**" Today a result
is a single `PASS` / `FAIL` / `NA` per (system, test[, container]). Evidence adds
a per-condition breakdown so an operator can see exactly which condition(s)
failed and what was expected.

---

## 2. Key insight

`compliance::evaluate()` already computes everything per condition and the
caller already collects per-condition `EvalResult`s before collapsing them with
`combine_verdict()`. So evidence is built **in the caller** from the
`(condition spec, EvalResult)` pairs it already holds — no change to the ~30
evaluator arms, so zero risk to evaluation logic.

---

## 3. Privacy — match-only, never raw content

**Decided:** evidence stores ONLY the admin-authored test spec (element,
parameter, sub-element, operator, expected value) + the per-condition verdict +
a **generic, content-free note**. It never carries host-observed content (file
bytes, command output, hashes). A condition that fails to match a file's
contents records `"expected to contains 'PermitRootLogin no' — not satisfied"`,
never the file itself. Secrets can't leak into the DB or a report.

`note` examples: `"matched"`, `"expected to equals 'enabled' — not satisfied"`,
`"applicability condition not met"`, `"not applicable / not evaluable on this
system"`.

---

## 4. Data model

`ConditionOutcome` (client, serialized to JSON):
```
phase     "condition" | "applicability"
element   FILE / SERVICE / OS / CONTAINER / …
input     the parameter (e.g. /etc/ssh/sshd_config)  ← admin-authored, safe
selement  sub-element
operator  contains / equals / >= / exists / …
expected  the admin-authored target                  ← safe
result    PASS | FAIL | NA
note      generic, content-free explanation
```

`build_evidence(phase, &[&TestCondition], &[EvalResult]) -> Option<String>`
zips conditions with their results and serialises the array. "All conditions,
flag failures": every condition is captured (≤10 per test), the UI highlights
the failing ones. Applicability misses are captured with `phase="applicability"`
so **NA is explained too**.

---

## 5. Flow

```
agent: evaluate() per condition (unchanged) → Vec<EvalResult>
       build_evidence(...) → JSON   ── alongside ──>  combine_verdict() → verdict
                                   │
ComplianceResult { …, evidence: Option<String> }   (serde-optional)
                                   │  POST /result
server: receive_result → INSERT … results.evidence (refreshed on conflict;
        exclusion columns still untouched)
                                   │
report: fetch_*_report_data loads r.evidence → IndividualResult.evidence
template: "why?" toggle on FAIL/NA rows → JS parses JSON → per-condition table
```

- Per-container results carry their own evidence (built from that container's
  results).
- `evidence` is `Option`/nullable end-to-end: pre-0.6.4 agents omit it, old
  servers ignore it, old report rows simply show no "why?" toggle.

---

## 6. Schema

`v31 → v32`: `results.evidence TEXT` (nullable). Idempotent column add; fresh
installs get it in `initialize_database`. Retention prune already bounds growth.

---

## 7. Scope notes / future

- Rendered in the **system report** and the **policy report** (both load
  `r.evidence`). Saved snapshots serialise `IndividualResult.evidence`, so
  reports archived after 0.6.4 keep their evidence.
- `runner.rs` (the local `scm test` debug runner) is unchanged — it prints to
  stdout and doesn't POST results, so server-side evidence doesn't apply.
- Possible later enrichment: include safe structured observed values (service
  state, version numbers) for non-content elements. Deliberately omitted in
  this cut to honour the strict match-only decision.
