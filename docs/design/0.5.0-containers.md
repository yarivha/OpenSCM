# Design — Container Support (0.5.0)

**Status:** approved, implementation pending
**Target release:** 0.5.0 (MVP) → 0.5.1, 0.5.2 (incremental) → 0.6.0 (Kubernetes)
**Author:** Yariv + Claude (design review)
**Last updated:** 2026-05-26

---

## 1. Goals

1. **Inventory** — every Linux host with the OpenSCM agent enumerates its running **app containers** (Docker, Podman) on each heartbeat. Results show as expandable rows under the host in the Systems list.
2. **Compliance scans on containers** — the existing CMD element gains a path that executes via `docker exec` / `podman exec`. A test in a policy targeting a host implicitly also runs (where applicable) against every container on that host.
3. **Container-only test elements** — a new family of elements (`PRIVILEGED`, `RUN_USER`, `IMAGE`, `MOUNT`, `NETWORK_MODE`, `EXPOSED_PORT`, `READ_ONLY_FS`, `HEALTH_CHECK`) that query container configuration metadata. Evaluated server-side from the inventory snapshot — no exec needed, fast.
4. **One canned CIS Docker policy** in `OpenSCM-store` to ship alongside the feature.

---

## 2. Critical mental model

> **"Containers" in OpenSCM = app containers** (Docker, Podman; later Kubernetes pods).
>
> **OS containers** (LXC, LXD, BSD jails) are full Linux userspaces with their own init systems. Install the OpenSCM agent **inside** them — they become regular systems with identical compliance surface. The host running them does **not** appear to have container inventory in OpenSCM, and that's correct.

This decision eliminates a lot of complexity that would otherwise surface in v1:

- No recursive discovery code (host enumerating LXC, then `lxc-attach` into each one to enumerate its inner Docker)
- No per-element compatibility matrices distinguishing OS-container from app-container test surfaces
- No `is_os_container` flag bifurcating the schema
- No question of "does this OS container run a real init? package manager? auditd?" — those are host questions answered by the inner agent

Nested case `VM → LXC → Docker` is solved by **two agents**: one on the VM (sees its own host stuff, no containers because LXC isn't an app container), one inside the LXC (sees its Docker containers as containers). Two systems, clean topology.

DinD with socket mount is not supported as a separate runtime — the inner containers appear on the outer Docker's inventory already because they share the daemon.

## 3. Data model

### New table `containers`

```sql
CREATE TABLE containers (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  tenant_id       TEXT    NOT NULL,
  host_system_id  INTEGER NOT NULL,           -- FK → systems.id
  runtime         TEXT    NOT NULL,           -- 'docker' | 'podman' (+ 'kubernetes' in 0.6.0)
  runtime_id      TEXT    NOT NULL,           -- native container/pod id
  name            TEXT    NOT NULL,
  image           TEXT,                       -- "nginx:1.27-alpine"
  image_digest    TEXT,                       -- "sha256:..."
  status          TEXT,                       -- 'running' | 'exited' | 'paused' | ...
  ip              TEXT,
  -- Metadata cached for container-only tests (all optional):
  is_privileged   INTEGER,
  run_user        TEXT,
  network_mode    TEXT,
  exposed_ports   TEXT,                       -- JSON array of "port/proto"
  mounts          TEXT,                       -- JSON array of {src,dst,type,ro}
  capabilities_add TEXT,                      -- JSON array (empty = defaults)
  read_only_fs    INTEGER,
  restart_policy  TEXT,
  health_check    INTEGER,                    -- 1 if HEALTHCHECK defined
  first_seen      DATETIME DEFAULT CURRENT_TIMESTAMP,
  last_seen       DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(host_system_id, runtime, name),
  FOREIGN KEY (host_system_id) REFERENCES systems(id) ON DELETE CASCADE
);

CREATE INDEX idx_containers_host  ON containers(host_system_id);
CREATE INDEX idx_containers_image ON containers(image);
```

### Existing `results` table extension

```sql
ALTER TABLE results ADD COLUMN container_id INTEGER;  -- NULL = host-level result
```

This gives per-container test history for free. The existing `results` aggregation logic uses `(system_id, test_id)` as a key; we add `container_id` as a third dimension.

### Lifecycle

- **Discovery is replace-on-heartbeat** — agent sends the full list of currently-running containers; server upserts by `(host_system_id, runtime, name)`, updates `last_seen`.
- **Stale aging** — daily prune deletes container rows where `last_seen < now - container_retention_days` (new setting, default `7`). Setting respects the existing retention pattern (`0` = forever).
- **Test results** retain `container_id` even after the container row is deleted — historical results survive container churn. Render gracefully ("container deleted") in archive views.

## 4. Discovery

### Detection

Agent runs `enumerate_containers()` once per heartbeat:

```
for runtime in [docker, podman]:
  if detect(runtime):
    for c in list_containers(runtime):
      enrich(c)
    payload[runtime] = list
```

| Runtime | Detection                              | Listing                                              |
|---------|----------------------------------------|------------------------------------------------------|
| Docker  | `/var/run/docker.sock` r/w OR `docker` CLI present | `docker ps -a --format '{{json .}}'`         |
| Podman  | `podman` CLI present                   | `podman ps -a --format json`                         |

Skip if the runtime isn't installed or the agent doesn't have access. Soft failure — don't fail the heartbeat.

### Enrichment

For each container, additionally run `inspect` to capture metadata:

```
docker inspect <id> -f '{{json .HostConfig}}{{json .Config}}'
```

Parse out: privileged, user, network mode, exposed ports, mounts, capabilities, read-only flag, restart policy, healthcheck presence.

### Heartbeat payload extension

```json
{
  ...,
  "containers": [
    {
      "runtime": "docker",
      "runtime_id": "a3f2...",
      "name": "nginx-prod",
      "image": "nginx:1.27-alpine",
      "image_digest": "sha256:...",
      "status": "running",
      "ip": "172.17.0.2",
      "is_privileged": false,
      "run_user": "nginx",
      "network_mode": "bridge",
      "exposed_ports": ["80/tcp", "443/tcp"],
      "mounts": [{"src":"/etc/nginx","dst":"/etc/nginx","type":"bind","ro":true}],
      "capabilities_add": [],
      "read_only_fs": false,
      "restart_policy": "always",
      "health_check": true
    }
  ]
}
```

All `containers` and its sub-fields are optional (`#[serde(default)]`) on both client and server. Old agents simply omit them.

### Permissions

Agent typically runs as root → Docker socket and Podman work transparently. For non-root deployments:
- Docker: agent user must be in the `docker` group
- Podman rootless: agent sees its own user's containers only (acceptable; document)

Documented in the new `containers.md` deployment guide. Not solved in v1.

## 5. UI

### Systems list

- New leftmost column with `▶`/`▼` chevron — only rendered for systems where `containers | length > 0`
- Click expands an inline nested table with rows showing: runtime icon, name, image, IP, status
- Container row click → side panel (or `/systems/{id}/containers/{cid}`) with full metadata + recent test results

Runtime badges: 🐳 Docker, 🦭 Podman, ☸ Kubernetes (when added).

### Container detail panel

| Section | Content |
|---|---|
| Identity | Name, runtime, ID, image, digest, IP, status, first/last seen |
| Configuration | All metadata fields (privileged, user, mounts, network, etc.) |
| Recent Test Results | Last N results scoped to this `container_id` |

No "approve" / "delete" actions — containers come and go on their own.

## 6. Tests on containers

### Element compatibility (0.5.0 MVP)

| Element        | Host | Container (0.5.0) | Container (later) |
|----------------|------|-------------------|-------------------|
| CMD            | ✓    | ✓ via `<rt> exec` | ✓                 |
| Container-only (new)  | NA  | ✓ from inventory metadata | ✓ |
| All others (FILE, PACKAGE, SERVICE, PROCESS, USER, GROUP, FIM, REGISTRY) | ✓ | **NA** in 0.5.0 | → 0.5.2 via probe-script or per-element shell-out |

This is the realistic MVP — anything beyond CMD-via-exec and metadata-only checks needs more design (probe binary upload, runtime image probing, etc.).

### CMD execution path

```rust
fn evaluate_cmd_in_container(c: &Container, input: &str, ...) -> EvalResult {
    let (exec, args) = match c.runtime.as_str() {
        "docker" => ("docker", vec!["exec", &c.runtime_id, "sh", "-c", input]),
        "podman" => ("podman", vec!["exec", &c.runtime_id, "sh", "-c", input]),
        _ => return EvalResult::Na,
    };
    let out = Command::new(exec).args(args).output();
    // existing apply_string_condition path
}
```

If the container has no shell (`scratch`, distroless), the exec fails and the test returns `FAIL` with a clear error in the result details. Acceptable v1 behaviour — document the limitation.

### Scoping — Model A (implicit inheritance)

- Policies remain scoped to system groups (existing model).
- For each system in scope, tests fan out across:
  1. The host itself (existing behavior)
  2. Every container on that host (new) — but only for the **container-compatible elements**
- Each test declares `target_type` in its JSON: `host` (default), `container`, or `both`. Drives which targets the test fires against.

This avoids a brand-new "container groups" entity and the associated UI/membership management. Containers being transient makes explicit groups a maintenance burden; filtering by `target_type` + image-name conditions covers the common cases.

### Tests with `target_type = container`

Per-test JSON schema gains:

```json
{
  "target_type": "container",          // or "host" (default), or "both"
  "container_runtime_filter": null,    // optional: "docker" / "podman" / regex
  "container_image_filter": null       // optional: glob/regex on image name
}
```

Both filters default to "match any". `container_image_filter: "nginx:.*"` would fire the test only against nginx containers.

## 7. Container-only elements

All eight are metadata-only — evaluated server-side from cached inventory. No agent round-trip needed for the test execution itself.

| Element          | Sub-elements                          | Source field                | Example use            |
|------------------|---------------------------------------|-----------------------------|------------------------|
| `IMAGE`          | NAME, TAG, DIGEST, REGISTRY           | image, image_digest         | "must be from internal registry" |
| `PRIVILEGED`     | EQUALS true/false                     | is_privileged               | CIS Docker 5.4         |
| `RUN_USER`       | EQUALS, NOT EQUALS                    | run_user                    | CIS Docker 4.1         |
| `MOUNT`          | EXISTS, NOT EXISTS                    | mounts                      | block `/var/run/docker.sock` mount |
| `NETWORK_MODE`   | EQUALS                                | network_mode                | CIS Docker 5.9         |
| `EXPOSED_PORT`   | EXISTS, NOT EXISTS, COUNT             | exposed_ports               | reduce attack surface  |
| `READ_ONLY_FS`   | EQUALS true/false                     | read_only_fs                | CIS Docker 5.12        |
| `HEALTH_CHECK`   | EXISTS                                | health_check                | observability          |

### Schema seed additions (migration to v26)

```
INSERT OR IGNORE INTO elements (name) VALUES
  ('IMAGE'), ('PRIVILEGED'), ('RUN_USER'), ('MOUNT'),
  ('NETWORK_MODE'), ('EXPOSED_PORT'), ('READ_ONLY_FS'), ('HEALTH_CHECK');

INSERT OR IGNORE INTO selements (name) VALUES
  ('NAME'), ('TAG'), ('DIGEST'), ('REGISTRY');
-- EXISTS/NOT EXISTS/COUNT/EQUALS already present
```

### Evaluation lives in `scmserver`, not the agent

Container-only elements query data the agent already shipped in the inventory. The compliance run can therefore evaluate them entirely server-side — no extra heartbeat round-trip. This is a slight architectural departure (today every test fires on the agent), but it's the right call for container metadata since the data is already centralized.

Pattern: `policy_run` checks each test's element type before dispatch — host-element tests go to the agent's queue as today, container-metadata tests evaluate inline on the server.

## 8. Retention

New per-tenant setting **`container_retention_days`** (default `7`). Container rows whose `last_seen` is older than the retention window are deleted by the daily prune task (same scheduler hook as audit / reports / notifications).

Test results scoped to a deleted container survive — render shows `(container removed)` in archive views.

## 9. MVP cut

### 0.5.0 — "containers exist and have metadata tests"

- Schema migration v25→v26 (containers table, results.container_id, 8 new elements + 4 new selements, container_retention_days setting)
- Agent: Docker + Podman discovery + enrichment, heartbeat payload extension
- Server: container ingest, retention prune, 8 container-only element evaluators
- UI: Systems list expand toggle, container detail panel
- One canned CIS Docker L1 policy in `OpenSCM-store` using the 8 elements (~15-20 tests)
- New `containers.md` page in OpenSCM-web

### 0.5.1 — "containers as scan targets"

- CMD element gets the `docker exec` / `podman exec` path
- Per-test `target_type` field (`host` / `container` / `both`) + `container_*_filter` fields
- Implicit inheritance: policies scoped to a host group also test the containers on those hosts

### 0.5.2 — "broader test coverage on containers"

- FILE / DIRECTORY / PACKAGE / USER / GROUP / PROCESS container variants
- Probe-script approach (agent uploads a small binary into the container once per scan) for syscall-style elements
- Document limitations for distroless/scratch images

### 0.6.0 — "Kubernetes"

- Kubeconfig handling per agent (RBAC, service-account tokens)
- Pod-level inventory (multi-container pods, init containers)
- Test scoping at pod or per-container granularity
- kubelet-API-only mode (single-node visibility without full cluster access)

## 10. Open questions for implementation phase

1. **Probe-script delivery format** — bundled with the agent binary (include_dir), or downloaded from server on first use? Affects how 0.5.2 ships.
2. **Container-result aggregation in reports** — group "host + its containers" together, or list each as a separate compliance subject? Drives report layout in 0.5.1.
3. **Container retention default** — `7` days feels right for "transient builder/CI workloads"; long-lived production containers wouldn't get pruned (still active). Confirm.
4. **Per-runtime detection failure visibility** — surface "Docker present but agent has no permission" as an admin warning, or just log it?

## 11. Non-goals (explicitly out of scope)

- Container compliance scoring rolling up into host compliance score — keep separate axes
- Container image vulnerability scanning — that's a separate roadmap item (#19)
- Image SBOM generation — also #20
- Container runtime hardening (testing the dockerd/podmand config) — already covered by host-level tests in `cis-docker-1.8.0-l1.json`
- Containerd / CRI-O standalone — they're underneath Docker/Podman/k8s; no separate inventory needed
- Windows containers — vanishingly rare in compliance contexts; revisit if customer asks
