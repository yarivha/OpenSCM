# Design — Golden Enrollment Tokens (0.6.0)

**Status:** approved, implementing
**Target release:** 0.6.0 (SaaS 0.4.0)
**Last updated:** 2026-06-02

---

## 1. Goal

Let an admin mint a **token** that auto-approves enrolling systems, removing the
manual "Approve" click for fleet provisioning. Drop the token into an agent's
config (image, cloud-init, config-management) and every system that enrolls
with it comes up **active** — auto-groups then place it and policies scan it,
fully hands-off.

---

## 2. Security model — approval bypass, NOT auth bypass

This is the single most important property. The token does **not** weaken
agent authentication:

- The agent still generates its ed25519 keypair and registers its public key.
- Every request is still signed and verified exactly as today.
- The token's *only* effect is to set a new system's initial `status` to
  `active` instead of `pending` — i.e. it skips the human approval step, the
  same thing an admin does by clicking Approve.

So a leaked token lets an attacker auto-approve a system *they could already
enroll as pending anyway* — it doesn't let them impersonate an existing system
or forge results. The blast radius is "systems show up active without review,"
which is bounded by expiry / max-uses / revocation (below).

---

## 3. Token format & storage

- Generated server-side: `oscm_` + 32 hex chars (16 random bytes). The `oscm_`
  prefix makes it greppable and recognisable in configs/logs.
- **Hashed at rest.** Only `SHA-256(token)` is stored (`token_hash`). The raw
  secret is shown **once** at creation and never again — exactly like a
  personal access token. A DB leak yields hashes, not usable tokens.
- A non-secret **display prefix** (`oscm_` + first 7 hex, e.g. `oscm_a1b2c3d`)
  is stored so the admin can identify a token in the list without exposing it.

## 4. Data model

```sql
CREATE TABLE enrollment_tokens (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  tenant_id    VARCHAR(191) NOT NULL DEFAULT 'default',
  name         TEXT NOT NULL,              -- admin label ("prod image", "lab")
  token_hash   TEXT NOT NULL,              -- SHA-256 hex of the secret
  token_prefix TEXT,                       -- non-secret display id
  enabled      INTEGER NOT NULL DEFAULT 1,
  expires_at   DATETIME,                   -- NULL = never expires
  max_uses     INTEGER,                    -- NULL = unlimited
  use_count    INTEGER NOT NULL DEFAULT 0,
  created_by   TEXT,
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  last_used_at DATETIME,
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE
);
CREATE INDEX idx_enrollment_tokens_lookup ON enrollment_tokens(tenant_id, enabled);
```

Schema **v30 → v31**.

A token is **valid** at enrollment time iff:
`enabled = 1` AND (`expires_at IS NULL` OR `expires_at > now`) AND
(`max_uses IS NULL` OR `use_count < max_uses`).

On a successful auto-approval the row's `use_count` is incremented and
`last_used_at` set, atomically inside the registration transaction.

## 5. Enrollment flow

```
agent config: server.enrollment_token = "oscm_…"
        │
        ▼  (first registration, id = 0)
  UnsignedPayload { …, enrollment_token: Some("oscm_…") }
        │
        ▼  /send  STEP 2 (id == 0)
  token present?
     │ no  → INSERT status = 'pending'   (unchanged behaviour)
     │ yes → validate_and_consume_token(tenant, token)
     │          valid   → INSERT status = 'active'
     │                    use_count++, last_used_at = now
     │                    audit "enrollment.token_approve"
     │          invalid → INSERT status = 'pending'  (LENIENT)
     │                    audit "enrollment.token_rejected" (warn)
```

- The token is sent **only at registration** (id = 0), never on subsequent
  heartbeats — it's not needed once the system has an ID.
- **Re-registration of an already-pending system**: if a system re-registers
  (same public key) while `pending` and presents a now-valid token, it is
  promoted to `active`. This covers "I enrolled before minting the token, then
  added it" without forcing a delete/re-enroll.
- **Lenient invalid handling** (decided): a typo'd/expired/disabled token never
  blocks enrollment — the system simply lands `pending` for manual approval,
  and the bad attempt is audited as a warning. No 403, no lost system.
- **Approval-only** (decided): the token does not assign groups. Placement is
  left to auto-groups, which sort the now-active system on its first heartbeat.

## 6. Admin UI (Systems page)

A token-management card on the Systems page (Admin-gated):

- **List** — name, display prefix, status (Active / Disabled / Expired),
  expiry, `use_count / max_uses`, last used, actions.
- **Create** — name + optional expiry date + optional max-uses → the raw
  secret is shown **once** in a copy-box with a "copy" button and a clear
  "you won't see this again" warning.
- **Disable / Enable** toggle and **Delete**.

Every create / disable / enable / delete is audited.

---

## 7. Agent config

New optional field under `[server]`:

```toml
[server]
url               = "https://scm.example.com"
organization      = "default"
enrollment_token  = "oscm_…"     # optional; only used at first registration
```

Windows registry: `EnrollmentToken` value under the server key.

The token lives in the agent config file, which is root/Administrator-readable
only (install sets perms). This is inherent to any join-token scheme (k8s, Salt,
Chef…) — the token is a provisioning secret, not a long-lived credential, and
expiry/max-uses bound its exposure.

---

## 8. Revocation

Disable or delete the token. Future enrollments presenting it fall back to
`pending`. Already-approved systems are unaffected (the token only ever gated
their initial status; their ongoing auth is the ed25519 signature).

---

## 9. Out of scope (v1)

- Per-token group assignment — auto-groups already handle placement.
- Single-use / one-time tokens — `max_uses = 1` covers that case.
- Rotating/rolling tokens — delete + mint a new one.
- Rate-limiting bad token attempts — the lenient path just audits; a flood of
  bad attempts is visible in the audit log.
